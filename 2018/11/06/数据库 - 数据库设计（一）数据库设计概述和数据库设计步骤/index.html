数据库 - 数据库设计（一）数据库设计概述和数据库设计步骤

发布日期 2018-11-06 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 3k | 阅读时间 ≈ 10

数据库设计概述

数据库的生命周期

  • 两个阶段：

    1. 数据库分析与设计阶段

      • 需求分析
      • 概念设计
      • 逻辑设计
      • 物理设计

    2. 数据库实现与操作阶段

      • 数据库的实现阶
      • 数据库的操作和监督阶段
      • 数据库的修改与调整阶段

数据库设计的目标

  • 两个目标：1. 满足应用功能 2. 良好的数据库性能

数据库设计的内容

  • 从用户对数据的需求出发，研究并构造数据库的过程
    1. 数据库的结构设计
    2. 数据库的行为设计

数据库设计的方法

  • 直观设计法

    特点：最原始的一类设计方法，利用设计者的经验和技巧设计数据库模式
    缺点：缺乏科学理论的指导，设计质量难以保证

  • 规范设计法

    1. 新奥尔良设计方法

    2. 基于 E-R 模型的数据设计方法

    基本思想：在需求分析的基础上用 E-R 图构造一个反映实现世界实体之间联系的企业模式，然后将此模式转换成某一特定DBMS 下的概念模式

    1. 计算机辅助设计方法

数据库设计过程

  1. 需求分析

  2. 结构设计

    • 概念结构设计
    • 逻辑结构设计
    • 物理结构设计

  3. 行为设计阶段

    • 功能设计
    • 事务设计
    • 程序设计

  4. 数据库实施阶段

    • 加载数据库数据
    • 调试运行应用程序

  5. 数据库运行和维护阶段

数据库设计的基本步骤

  1. 使用的方法：分阶段设计方法

  2. 原则：自顶向下，逐步求精的原则

    将数据库设计过程分解为若干个相互依存的阶段，每一阶段采用不同的技术与工具解决不同的问题

    目的：将问题局部化，减少 局部问题对整体设计的影响，便于多人合作

数据库设计的各个阶段，又称为数据库设计步骤

1. 需求分析

  • 目标：了解与用户信息及应用处理的要求，并将结果按一定格式整理而形成需求分析报告

    该报告是后续概念设计，逻辑设计，物理设计，数据库建立与维护的依据

  • 4 个步骤：

    1. 确定数据库范围

      • 目的：确定数据库应支持哪些应用功能

      • 首先要考虑得是支持用户工作需要所必须的应用要求

    2. 分析数据应用过程

      • 目的：了解分析数据与数据处理间的关系

      • 每个功能用到那些数据，数据使用顺序，对数据作如何处理以及处理的策略以及处理的结果

      • 应用过程分析的结果是数据库结构设计的重要依据

        1. 那些数据需要存入数据库中
        2. 哪些数据可作为报告输出而不存入数据库中
        3. 不同数据的不同使用权限（存，取，删，改） 和共享范围

      • 数据应用过程也是数据库应用程序编写的依据

      • 使用的工具

        1. 借助数据流程图或其他信息及应用结构图形表示
        2. 在构建数据流程图的时候需要注意：
          • 应以局部应用部门或相对独立的应用功能为标识单位，从而将问题局部化，有利于分块设计

    3. 收集与分析数据

      • 目的：了解并分析数据的组成格式以及操作特征，每个数据元素的语义及关系等，并将他们收集起来整理归档。

        数据流程图中所涉及的 数据都是收集与分析的对象，主要包括报表，文件，单据以及各种原始资料

      • 可以从三个方面入手：

        1. 数据的静态结构

          • 概念：指不施加应用操作于其上时数据的原始状态，可以通过数据分类表和数据元素进行说明
          1. 数据分类表： 用于数据的总体描述，对于每一客观存在的具有独立意义的单类数据单位（如一种报表，一种文件，一种收据）应给出其名称，用途，编制者以及使用者等说明

          • 数据分类表，其结构

          • 数据ID

          • 数据名
          • 用途
          • 主人
          • 用户
          • 来源
          • 去向
          • 存档时间

          • 数据量

          • 数据元素表，其结构

          • 数据ID

          • 数据元素ID
          • 元素名
          • 意义
          • 类型
          • 长度
          • 算法
          • 备注

        2. 动态结构

          • 概念：指将应用操作施加于数据之上数据的状态，可通过任务分类表和数据操作特征表进行说明
          • 意义：
          • 任务分类表
          • 任务 ID
          • 名称
          • 功能
          • 类型
          • 主人
          • 用户
          • 执行日期
          • 频率
          • 操作过程

          • 备注

          • 数据操作特征表

          • 任务 ID
          • 数据 ID
          • 建立
          • 查询
          • 插入
          • 修改
          • 删除
          • 数据量

        3. 数据约束

          • 数据约束是指使用数据时的特殊要求，约束的类型：

          1. 数据的安全保密性

            谁拥有操作，存，取，删，改的不同权限

          2. 数据的完整性

            数据正确性的约束范围和验证准则

          3. 响应时间

            指主要是指某些特定应用要求的数据存储时间限制

          4. 数据恢复

            其主要指转储及恢复的时机与范围等要求

    4. 编写需求分析报告

      • 作为需求分析阶段的一个总结，需求分析报告作为应用需求的业务人员和数据库设计人员的 “共同语言”

      • 需求报告的内容：

        1. 数据库的应用功能目标

          要求明确标明数据库的应用范围以及达到的应用处理功能

        2. 标明不同用户视图范围

          根据机构与职能关系图和数据流程图，并参考任务分类表等，确定不同部门或功能的局部视图范围

        3. 应用处理过程需求说明

          1. 数据流程图，其主要反映应用部门原始业务处理的工作流程
          2. 任务分类表，其标明不同任务的功能以及使用状况
          3. 数据操作特征表，标明任务和数据间的联系以及不同数据的不同操作特征与执行效率

        4. 数据字典

          • 数据字典（Data Dictionary, DD）概念：数据库系统中存储三级结构定义的数据库

            通常指的是数据库系统中各类数据详细描述的集合

          • 功能：存储和检索各种数据描述，即元数据

          • 特点：在数据库设计中，它提供了对各类数据描述的集中管理，是一种数据分析，系统设计和管理的有力工具

            数组字典需要有专人进行管理，及时对数据字典进行更新，保证字典的安全，可靠

          • 数据字典包括的内容：数据分类表，数据元素表和各类原始资料

            各类原始资料是指收集所有单据，报表，文件以及设计所需的原始资料，并根据数据分类表的数据 ID 统一分类标号

2. 概念结构设计

  • 目的（任务）：是在需求分析中产生的需求分析报告的基础上，按照特定的方法设计满足应用需求的用户信息结构

    改信息结构通常称为概念模式

  • 特点：概念模型独立于任何软件和硬件，

    主要目的：是最大限度地满足应用需求，可完全不顾及具体的硬件和软件的限制，因而它是一个符合用户需求的趋于理想化的信息结构

  • 概念结构设计的常用方法：

    1. 实体分析法（自顶向下法）
    2. 属性综合法（自底向上法）

      采用属性分析开始，高层实体以及联系通过底层属性组成的设计技术

      实际上，这是一种基于统计分析推导的方法，即通过对数据元素与应用任务联系的定性定量统计分析技术来推导的方法

      统计分析推导的处理过程：可分为属性分类，实体构成，联系的确定等相对独立的步骤

3. 逻辑结构设计

  • 目的（任务）：是将概念模型转化为等价的，并为特定 DBMS 所支持数据模型的结构

  • 数据库逻辑模型表示方法：层次，网状，关系数据模型表示

  • 逻辑结构设计的输入和输出信息

    逻辑结构设计是在需求分析以及概念结构设计的基础上进行的，通常要求提供如下输入信息

    1. 独立于特定 DBMS 的概念模型
    2. 有关响应时间，安全保密性，数据完整性及恢复方面的要求说明，包括保持数据一致性的规则说明
    3. 数据量以及使用频率
    4. 特定 DBMS 特性，包括 DBMS 支持的数据模型以及数据定义语言的说明

  • 完成逻辑结构设计之后输出的信息

    1. 一个特定 DBMS 支持的概念模式，或称为模式
    2. 物理设计说明，主要包括存入数据库中的数据量，使用频率以及响应时间要求
    3. 程序设计说明，其是在需求分析的基础上，根据已完成的逻辑模型编制各程序名，执行逻辑步，存取数据的名，顺序以及操作特征（读写删改）的说明

  • 逻辑结构设计的步骤

    1. 模型转换是指将概念模型等价地转换为特定 DBMS 支持的关系模式，网状模型或层次模型

    2. 子模式设计的目标：是抽取或导出模式的子集，以构造不同用户使用的局部数据逻辑结构

    3. 编制应用程序设计说明的目的：为可实际运行的应用程序设计提供依据与指导，并作为设计评价的基础

    4. 设计评价的任务是分析并检验模式以及子模式的正确性合理性

4. 物理结构设计

  • 概念：是指对于一个给定的数据库逻辑结构，研究并构造物理结构的过程
  • 任务：是确定数据库在存储设备上的存储结构以及存储方法

    因 DBMS 的不同还可能包括建立索引和聚集，以及物理块大小，缓冲区个数和大小，数据压缩的选择等

    对于物理结构设计的人员的要求

    • 掌握基本存储设备的特性
    • 充分了解操作系统以及 DBMS 所提供的支持环境
    • 并十分熟悉数据库的逻辑结构以及存取速度，频率以及数据量方面的要求

5. 数据库实施

  • 工作内容：加载数据，应用程序设计以及数据库运行

  • 数据加载

    概念：将符号要求的数据装载到数据库中去，其具体过程包括数据的收集，分类，整理校验和输入等过程，经此就可将数据按数据库定义准确地存储到数据库中

  • 应用程序设计

    严格说，应用程序设计不应是数据库设计的范畴，但它又必须且只能是随着数据库结构设计的进展而逐步地并行进行

  • 数据库试运行

    目的：为了避免对实际应用带来损害，数据库通常应经过一个试运行阶段

6. 数据库的运行和维护

  • 数据库系统投入实际运行标志着数据库设计和应用开发的基本完成，但绝不是意味着设计应用开发工作的终止
