数据库系统的结构

数据库系统的分类

  • 数据库管理员（DBA）的角度

    1. 内部系统

      • 通常采用三级模式结构

    2. 外部系统

      • 通常采用集中式结构，分布式结构和并行结构等

  • 从用户的角度

    1. 客户/服务器结构
    2. 浏览器/服务器结构

  • 数据库系统的三级模式结构

    • 三级模式结构（采用三级模式结构，并提供两层映像功能）：

      1. 用户级
      2. 概念级
      3. 物理级

    • 具体：

      1. 模式
      2. 外模式
      3. 内模式

模式

  • 模式也称为概念模式或逻辑模式

    它是数据库中全体数据的逻辑结构和特征的描述，是所有用户的功能数据视图

  • 模式实际上市数据库在逻辑级上的视图，概念视图

    特点：

    1. 其形式要比数据的物理存储方式抽象

    2. 不包含任何数据库的实现细节，而是定义数据的逻辑结构，数据之间的联系，以及数据相关的安全性，完整性(DBA 必须考虑整个数据库的这种全局概念视图)

    3. 处于数据库外模式和内模式的中间，一个数据库只有一个模式

外模式

  • 三级模式结构中，外模式也称为子模式（Subschema）或用户模式
  • 是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据和逻辑结构和特征的描述
  • 是某一应用有关的数据的逻辑表示
  • 实际上：用于满足不同数据库用户需求的数据视图，即用户视图。通常是模式的子集，是对数据整体数据结构的局部重构
  • 一个数据库可以有多个不同的外模式，允许它们有一定的重叠、

    DBMS 提供子模式描述语言（Subschema DDL）来严格定义子模式

内模式

  • 在三级模式结构中，内模式也称为存储模式（Storage Schema）
  • 它是对数据库中数据物理结构和存储方式的描述，是数据在数据库内部的表示形式
  • 实际上：是整个数据库的最底层表示，它不同于物理层，是 DBA 所见的，特定的 DBMS 所处理的数据库的内部结构，即内部视图或存储视图
  • 一个数据库只用一个內模式

    內模式不是存储设备上的物理记录或物理块，也不涉及具体设备限制，内模式还不是最底层的物理层，比内模式更接近物理存储和访问的那些软件机制是操作系统的一部分

    DBMS 提供內模式描述语言（內模式 DDL, 或存储模式 DDL）来严格定义内模式

    1. 定义内部记录类型，索引和文件的组织方式
    2. 以及所有数据控制方面的细节（例如：记录存储方式是按照顺序结构存储还是按照 B 树结构存储）
    3. 数据是否压缩，加密等

三级模式结构的两层映像与数据独立性

  • 数据库系统三级模式结构的三个模式分别是对数据的三级抽象，它们之间的关系

    特点：

    1. 一个数据库的整体逻辑结构和特征的描述（概念模式）是独立于数据库其他层次结构（内/外模式）的描述，是数据库的核心，也是数据库设计的核心
    2. 一个数据库的内部存储模式依赖于概念模式，但存储模式独立于外部模式，也独立于具体的存储设备
    3. 用户逻辑结构（外模式）是在全局逻辑结构描述的基础上定义的，它面向具体的应用程序，独立于内部模式和存储设备
    4. 特定的应用程序是在外模式的逻辑结构上编写，它依赖于特定的外模式，与数据库的模式和存储结构独立
  • 为了有效支持数据的三级抽象以及他们相互间的联系和转换，DBMS 通过在内部提供三级模式之间的两层映像来实:
    1. 外模式/模式映像
    2. 模式/内模式映像
外模式/模式映像
  1. 同一个模式可以有任意多个外模式
模式/内模式映像
  1. 模式/内模式映像是唯一的

  • 数据库是什么

  • 数据库管理系统（DBMS）是什么

    2. 是专门用于建立和管理数据库的一套软件，介于应用程序和操作系统之间

    3. 功能：
      1. 数据定义功能

      3. 数据库运行管理功能

视图和表的关系

  • 从 SQL 的角度上看，一个视图就是张表
  • 视图和表的区别只有一个：是否保存实际的数据
    1. 表是上的数据实际上保存到计算机中的硬盘中（通常）
    2. 视图则不同，它不会将数据保存到其他任何地方
  • 使用视图的优点：
    1. 可以节省存储设备的空间
    2. 将频繁使用的复杂的 SELECT 语句保存成视图，操作方便

事务

  • 事物就是：需要在同一个处理单元中执行的一系列更新处理的集合

    同一处理单元：同一个数据库，同一个表中

  • 创建事务(MySQL)

    SQL 标准中是没有规定开始指令，根据 DBMS 的不同而不同，实际上几乎所有的数据库产品无需开始指令
    上面的是 MySQL 事务的操作

  • 事物的区分，有两种情况

    1. 每一条 SQL 语句就是一个事务（自动提交模式）
    2. 知道用户执行 COMMIT 或者 ROLLBACK 为止算作一个事物

  • 例子：

    上面的例子发生的事情：

    1. 创建了 ProductTest TABLE ,并设置 列的配置
    2. 使用 START TRANSACTION 语句告诉 MySQL 数据库我们开始一个事务
    3. 插入一条数据
    4. 做一次 ProductTest TABLE 的查询
    5. 这里示范 ROLLBACK ,取消事务的操作
    6. 最后一句查询 ProductTest TABLE 是验证 ROLLBACK 是否生效

    如果 不使用 ROLLBACK ，而是 COMMIT 操作就不一样了，插入的数据就会更新到 ProductTest TABLE 中

事务的特性（ACID）

  • 原子性(ATOMICITY)：每一条 SQL 语句都会按顺序执行
  • 一致性(Consistency)：出错的语句不会被执行，但是不会影响后面的语句（除非上一句 SQL 语句是下一句 SQL 语句的铺垫）
  • 隔离性(Lsolation)：事务之间是不可见的
  • 持久性(Durability)：在事务结束后改时间点的数据状态会被保存的特性

    保证持久性的方法：常见的就是将事务的执行记录保存到硬盘等介质中（该执行记录称为 ”日志“）
