数据库 - SQL 和关系型数据库的基本操作

发布日期 2018-11-12 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 673 | 阅读时间 ≈ 2

SQL 概述

SQL 发展

  1. SQL 是于 1974 年由 Boyce 和 Chamberlin 提出，并在System R（IBM 公司研制的RDMS）
  2. SQL 标准是 1986 年 美国国家标准局 (American National Standard Institute, ANSI)颁布
  3. 1987 年 6 月 国际化标准组织（International Organization for Standardization, ISO）正式采纳它为国际标准
  4. 目前 SQL 的标准是 SQL:2018
  5. 数据库管理系统产品：MySQL, Oracle, SQL Sever 等

SQL 的特点

  1. SQL 不是某个特定数据库供应商专有的语言
  2. SQL 简单易学（语句全部是具有很强的描述性的英语单词组成，而且数目不多）
  3. SQL 看上去很简单，灵活使用其语言元素，可以进行非常复杂和高级的数据库操作

    SQL 语句不区分大小写

SQL 的组成

  • SQL 的主要功能：1. 数据定义，2. 数据操纵（数据查询），3. 数据控制

    1. 数据定义语言（Data Definition Language, DDL）
    • 主要功能：对数据库以及数据库中的各种对象进行创建，删除，修改等操作
    • 数据库对象：表，默认约束，规则，视图，触发器，存储过程等
    • DDL 主要包括 SQL 语句以下三种：
      1. CREATE: 用于创建数据库或数据库对象
      2. ALTER: 用于对数据库或数据库对象尽心修改
      3. DROP: 用于删除数据库或数据库对象
    1. 数据操作语言（Data Manipulation Language, DML）
    • 主要功能: 主要用于操作数据库中各种对象，特别是检索和修改数据
    • DML 主要包括 SQL 语句以下几种：
      1. SELECT: 用于从表中或视图中检索数据，其是数据库中使用最为频繁的 SQL 语句
      2. INSERT: 用于将数据插入到表或视图中
      3. UPDATE: 用于修改表或视图中的数据，其既可修改表或视图中的一行数据，也可以同时修改多行或全部数据
    1. 数据控制语言（Data Control Language, DCL）
    • 主要功能: 主要用于安全管理，例如确定哪些用户查看或修改数据库中的数据
    • DCL 主要包括 SQL 语句以下几种
      1. GRANT: 用于授予权限，可把语句许可或对象许可的权限授予其他用户和角色
      2. REVOKE: 用于收回权限，其功能与 GRANT 相反，但不影响该用户或角色从其他角色中作为成员继承许可权限

数据库 - 数据库设计（二）关系数据库设计方法

发布日期 2018-11-10 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 3.1k | 阅读时间 ≈ 10

关系数据库库设计的过程概述

  • 关系数据库：一类采用关系模型作为逻辑数据模型的数据库系统
  • 关系数据库设计的过程遵循数据库设计的基本步骤
    1. 需求分析
    2. 概念结构设计
    3. 逻辑结构设计
    4. 物理结构设计
    5. 数据库实施
    6. 数据库的运行和维护等阶段

关系数据库设计过程与各级模式

  1. 需求分析阶段：综合各个用户的应用需求
  2. 在概念结构设计阶段：形成独立于机器特点，独立于各个关系数据库管理系统产品（MySQL,Oracle,SQL,Server 等）的概念模式，如使用 E - R 图
  3. 在逻辑结构设计阶段，将 E - R 图转换成具体的数据库产品支持的关系数据模型，形成数据库逻辑模式，然后根据用户处理的要求，安全性考虑，在基本表的基础上再建立必要的视图，形成数据的外模式
  4. 再物理结构的设计阶段中，根据关系数据库管理系统的特点和处理的需要，进行物理存储安排，建立索引，形成数据库内模式

概念结构设计方法

  1. 关系数据库的概念结构设计方法通常是：自顶向下法

  2. 两个步骤

    • 建立局部信息结构
    • 然后将局部信息结构合成全局信息结构并优化
  3. 描述工具：
    • E - R 图

概念结构设计方法常使用的描述工具：E - R 图的表示法

  1. 概念结构设计：是将需求分析得到的用户需求抽象为信息结构（概念模型）的过程，它通常使用 E - R 图作为描述现实世界的建模工具

  2. E - R 图提供了表示信息世界中实体，属性和联系的方法，具体：

    • 实体型：其用矩形表示，矩形框内写明实体的名称
    • 属性：其用椭圆形表示，并用无向边将其与相应的实体链接起来
    • 联系：其用菱形表示，菱形框内写明联系的名称，并用无向边分别与有关实体连接起来，同时在无向边旁标上联系的类型（1:1,1:N, M:N），如果一个联系具有属性，则这些属性也要用无向边与该联系连接起来

  3. 两个实体型之间的联系：

    • 一对一（1:1）
    • 一对多（1:N）
    • 多对多（M:N）

  4. 两个以上的实体型之间的联系

    • 一对一（1:1）
    • 一对多（1:N）
    • 多对多（M:N）

  5. 单个实体型内的联系

    • 一对一（1:1）
    • 一对多（1:N）
    • 多对多（M:N）

概念结构设计方法第一个步骤：局部信息结构设计

  • 概念：根据需求分析报告中标明的用户视图范围所建立的满足该范围内用户需求的信息结构

  • 局部信息结构设计的步骤分为：

    1. 确定局部范围
    2. 选择实体
    3. 选择实体关键字
    4. 确定实体间联系
    5. 确定实体的属性

局部信息结构设计第一个步骤：确定局部范围

  • 局部范围主要依据需求分析报告后者难过标明的用户视图范围来确定

    需求分析报告中标明的局部范围往往与子模式范围相对应

    范围确定的基本准则:

    1. 部门和功能相对对立
    2. 同其他局部范围相互影响较少，并且实体个数适量

局部信息结构设计第二个步骤：选择实体

  1. 选择实体的（目的）任务：是在确定的局部范围内选择一些合适的信息单位作为局部信息结构的基本实体

    这些实体应能满足该局部范围内的应用需求
    通常，以实际工作中习惯使用的信息单位作为选择实体的基础，数据分类表是选择实体的直接依据

    再同一局部信息结构中的实体名称应具有唯一性

  2. 实体选择时的最大困难是如何区别实体与属性

局部信息结构设计第三个步骤：选择实体的关键字属性

  • 实体的存在依赖于其他关键字的存在

    实体关键字属性确定之后，实体的非关键字属性也就容易确定了，因为他们之间存在某种应用上的联系和依赖性

局部信息结构设计第四个步骤：确定实体间的联系

  • 数据间的联系必须在概念设计时确定

    一种常用的分析实体间是否存在联系的方法是，将局部范围内的实体逐一取出来与该范围内的其他实体试行匹配（在任务表中）

局部信息结构设计第五个步骤：确定实体的属性

  • 属性分为两类

    1. 标识属性
    2. 说明属性

  • 标识属性（实体关键字属性和非关键字属性）

    用作实体的关键字，说明属性用作描述实体的一般特征

  • 说明属性

    基本原则：说明属性的存在和使用通常依赖于标识属性
    取决于：该属性是否可通过其所在实体的关键字找到，并且他们之间在应用中具有某种联系

    说明属性应是单值的，不允许嵌套属性和重复组的现象出现在实体中
    会有这种情况：同一说明属性依赖于多个实体的标识属性的情况，此时可将该属性分配到其他使用频率最高的实体中去

概念结构设计方法第二个步骤：全局信息结构设计

  • 概念：是将上述步骤中产生的所有局部信息结构合并成一个全局信息结构

  • 全局信息结构必须是所有局部信息结构的全面准确的映像

    即合并前各局部信息结构能实现的应用需求，合并之后也能实现

  • 需要避免的问题：消除不同局部信息结构中因用户观点的不同所可能导致的数据不一致，并尽可能地增强数据的共享性，控制数据的冗余

  • 合并的前提：假设局部信息结构都是完全的，一致的前提下进行

    即假设各局部信息结构都能满足其对应局部范围的应用需求,且其内部不存在需要合并的成分

  • 合并的方式

    1. 可以一次性把所有的局部信息结构合并成全局的信息结构
    2. 也可以采用逐步合并，进行累加的方式

  • 合并的过程

    1. 一个系统中有多个功能，可能有多个设计人员进行设计，合并的时候就会出现冲突，这个无法避免的，需要合理的解决

  • 合并中出现的冲突

    1. 属性冲突

      • 属性域冲突

        属性值的类型，取值范围，取值集合的不同

      • 属性取值单位冲突

    2. 命名冲突，两类

      • 同名异义
      • 异名同义

    3. 结构冲突，三类

      • 同一对象在一局部 E - R 图中作为实体，而在另外一个局部 E - R 图中作为属性
      • 同义实体在不同的 E - R 图中属性个数和类型不同
      • 实体之间的联系在不同的 E - R 图中是不同的类型

    4. 好的全局信息结构设计

      1. 能反应用户功能需求
      2. 实体的类型数量尽可能少
      3. 实体类型所含属性尽可能少
      4. 实体类型间联系无冗余

        后面三个内容，是全局 E - R 模型优化需要达到内容
        优化步骤：

        1. 相关实体类型的合并
        2. 消除不必要的冗余属性
        3. 消除不必要的冗余联系
        4. 有时为了效率，可以适当的存在冗余（😂你懂的）

逻辑结构设计方法

  • 在数据库设计步骤中逻辑结构的设计

    1. 目的（任务）：是把在概念结构设计产生的概念模型转换为具体的 DBMS 所支持的逻辑数据模型

      导出特定的 DBMS 可以处理的数据逻辑结构（数据库的模式和外模式），这些模式在功能，性能，完整性和约束性方面满足应用要求

  • 在关系型数据库的设计中逻辑结构的设计：

    1. 目的（任务）：就是把概念结构设计阶段已设计好的 E - R 图转换为关系数据库管理系统所支持的关系模型
    2. 工作内容：
      • 将 E - R 图转换位关系模型
      • 对关系数据库模型进行优化
      • 设计面向用户的外模式

逻辑结构设计内容（一）：将 E - R 图转换位关系模型

  • 需要解决的问题：

    1. 要如何将实体以及实体间的联系转换为关系模型
    2. 如何确定这些关系模式的属性和主码

      关系模式的逻辑结构是一组关系模式的集合，而 E - R 图则是由实体，实体的属性和实体间的联系转换为某种关系模式

  • 原则：(待更新)

    1. 一个实体转换为一个关系模式
    2. 一个一对一（1:1）联系可以转换为一个独立的关系模式
    3. 一个一对多（1:N）联系可以转换为一个独立的关系模式
    4. 一个多对多（M:N）联系转换为一个关系模式
    5. 三个或桑哥以上实体间的一个多元联系可以转换为一个关系模式

逻辑结构设计内容（二）：对关系数据库模型进行优化

  • 数据库逻辑设计的结果不是唯一的
  • 数据模型的优化： 为了进一步提高数据库的要求，还应该根据应用需要适当修改地修改，调整数据模型额结构
  • 关系数据模型的优化通常以关系规范化理论为指导，数据库 - 关系型数据库（三）关系数据库的规范化理论

逻辑结构设计内容（三）：设计面向用户的外模式

  • 将概念模式转化为全局逻辑模式之后，可根据局部应用需求，利用视图（View）设计更符合局部用户需要的用户外模式
  • 定义数据库的全局模式主要是从系统的时间效率，空间效率，易维护性等角度出发。由于用户外模式与模式是相对独立的，所以定义用户外模式时注意：
    1. 定义别名
    2. 对应不同级别的用户定义不同的视图，以保证系统的安全性
    3. 简化用户对系统的使用

物理设计方法

  • 关系数据库系统的优点之一是用户通常不需要进行数据存储结构和存取方法的是设计

    这个是相对的，用户失去对数据库存取路径的控制而影响到数据库的存取效率

  • 物理设计的（目的）任务：主要是通过对关系建立索引和聚集来实现与应用相关数据的逻辑连接和物理聚集，以改善对数据库的存取效率

物理设计方法（一）： 建立索引

  • 建立索引的方式通常有静态和动态

物理设计方法（二）：建立聚集

  • 聚集：是将相关数据集中存放的物理存储技术

    以提高 I/O 的数据命中率而改善存取速度，其功能由具体的 DBMS 所提供

    所谓集中存放的概念：将相关数据尽可能地存放一个物理块中（或一个磁道中，或一个柱面中，或相邻区域块，磁道，柱面）

  • 数据聚集结构的一种有效方式是块结构方式。

    块域块之间由指针连接，一个块对于一个物理分区

    数据聚集可在一个或多个关系上建立

数据库 - 数据库设计（一）数据库设计概述和数据库设计步骤

数据库设计概述

数据库的生命周期

  • 两个阶段：

    1. 数据库分析与设计阶段

      • 需求分析
      • 概念设计
      • 逻辑设计
      • 物理设计

    2. 数据库实现与操作阶段

      • 数据库的实现阶
      • 数据库的操作和监督阶段
      • 数据库的修改与调整阶段

数据库设计的目标

  • 两个目标：1. 满足应用功能 2. 良好的数据库性能

数据库设计的内容

  • 从用户对数据的需求出发，研究并构造数据库的过程
    1. 数据库的结构设计
    2. 数据库的行为设计

数据库设计的方法

  • 直观设计法

    特点：最原始的一类设计方法，利用设计者的经验和技巧设计数据库模式
    缺点：缺乏科学理论的指导，设计质量难以保证

  • 规范设计法

    1. 新奥尔良设计方法

    2. 基于 E-R 模型的数据设计方法

    基本思想：在需求分析的基础上用 E-R 图构造一个反映实现世界实体之间联系的企业模式，然后将此模式转换成某一特定DBMS 下的概念模式

    1. 计算机辅助设计方法

数据库设计过程

  1. 需求分析

  2. 结构设计

    • 概念结构设计
    • 逻辑结构设计
    • 物理结构设计

  3. 行为设计阶段

    • 功能设计
    • 事务设计
    • 程序设计

  4. 数据库实施阶段

    • 加载数据库数据
    • 调试运行应用程序

  5. 数据库运行和维护阶段

数据库设计的基本步骤

  1. 使用的方法：分阶段设计方法

  2. 原则：自顶向下，逐步求精的原则

    将数据库设计过程分解为若干个相互依存的阶段，每一阶段采用不同的技术与工具解决不同的问题

    目的：将问题局部化，减少 局部问题对整体设计的影响，便于多人合作

数据库设计的各个阶段，又称为数据库设计步骤

1. 需求分析

  • 目标：了解与用户信息及应用处理的要求，并将结果按一定格式整理而形成需求分析报告

    该报告是后续概念设计，逻辑设计，物理设计，数据库建立与维护的依据

  • 4 个步骤：

    1. 确定数据库范围

      • 目的：确定数据库应支持哪些应用功能

      • 首先要考虑得是支持用户工作需要所必须的应用要求

    2. 分析数据应用过程

      • 目的：了解分析数据与数据处理间的关系

      • 每个功能用到那些数据，数据使用顺序，对数据作如何处理以及处理的策略以及处理的结果

      • 应用过程分析的结果是数据库结构设计的重要依据

        1. 那些数据需要存入数据库中
        2. 哪些数据可作为报告输出而不存入数据库中
        3. 不同数据的不同使用权限（存，取，删，改） 和共享范围

      • 数据应用过程也是数据库应用程序编写的依据

      • 使用的工具

        1. 借助数据流程图或其他信息及应用结构图形表示
        2. 在构建数据流程图的时候需要注意：
          • 应以局部应用部门或相对独立的应用功能为标识单位，从而将问题局部化，有利于分块设计

    3. 收集与分析数据

      • 目的：了解并分析数据的组成格式以及操作特征，每个数据元素的语义及关系等，并将他们收集起来整理归档。

        数据流程图中所涉及的 数据都是收集与分析的对象，主要包括报表，文件，单据以及各种原始资料

      • 可以从三个方面入手：

        1. 数据的静态结构

          • 概念：指不施加应用操作于其上时数据的原始状态，可以通过数据分类表和数据元素进行说明
          1. 数据分类表： 用于数据的总体描述，对于每一客观存在的具有独立意义的单类数据单位（如一种报表，一种文件，一种收据）应给出其名称，用途，编制者以及使用者等说明

          • 数据分类表，其结构

          • 数据ID

          • 数据名
          • 用途
          • 主人
          • 用户
          • 来源
          • 去向
          • 存档时间

          • 数据量

          • 数据元素表，其结构

          • 数据ID

          • 数据元素ID
          • 元素名
          • 意义
          • 类型
          • 长度
          • 算法
          • 备注

        2. 动态结构

          • 概念：指将应用操作施加于数据之上数据的状态，可通过任务分类表和数据操作特征表进行说明
          • 意义：
          • 任务分类表
          • 任务 ID
          • 名称
          • 功能
          • 类型
          • 主人
          • 用户
          • 执行日期
          • 频率
          • 操作过程

          • 备注

          • 数据操作特征表

          • 任务 ID
          • 数据 ID
          • 建立
          • 查询
          • 插入
          • 修改
          • 删除
          • 数据量

        3. 数据约束

          • 数据约束是指使用数据时的特殊要求，约束的类型：

          1. 数据的安全保密性

            谁拥有操作，存，取，删，改的不同权限

          2. 数据的完整性

            数据正确性的约束范围和验证准则

          3. 响应时间

            指主要是指某些特定应用要求的数据存储时间限制

          4. 数据恢复

            其主要指转储及恢复的时机与范围等要求

    4. 编写需求分析报告

      • 作为需求分析阶段的一个总结，需求分析报告作为应用需求的业务人员和数据库设计人员的 “共同语言”

      • 需求报告的内容：

        1. 数据库的应用功能目标

          要求明确标明数据库的应用范围以及达到的应用处理功能

        2. 标明不同用户视图范围

          根据机构与职能关系图和数据流程图，并参考任务分类表等，确定不同部门或功能的局部视图范围

        3. 应用处理过程需求说明

          1. 数据流程图，其主要反映应用部门原始业务处理的工作流程
          2. 任务分类表，其标明不同任务的功能以及使用状况
          3. 数据操作特征表，标明任务和数据间的联系以及不同数据的不同操作特征与执行效率

        4. 数据字典

          • 数据字典（Data Dictionary, DD）概念：数据库系统中存储三级结构定义的数据库

            通常指的是数据库系统中各类数据详细描述的集合

          • 功能：存储和检索各种数据描述，即元数据

          • 特点：在数据库设计中，它提供了对各类数据描述的集中管理，是一种数据分析，系统设计和管理的有力工具

            数组字典需要有专人进行管理，及时对数据字典进行更新，保证字典的安全，可靠

          • 数据字典包括的内容：数据分类表，数据元素表和各类原始资料

            各类原始资料是指收集所有单据，报表，文件以及设计所需的原始资料，并根据数据分类表的数据 ID 统一分类标号

2. 概念结构设计

  • 目的（任务）：是在需求分析中产生的需求分析报告的基础上，按照特定的方法设计满足应用需求的用户信息结构

    改信息结构通常称为概念模式

  • 特点：概念模型独立于任何软件和硬件，

    主要目的：是最大限度地满足应用需求，可完全不顾及具体的硬件和软件的限制，因而它是一个符合用户需求的趋于理想化的信息结构

  • 概念结构设计的常用方法：

    1. 实体分析法（自顶向下法）
    2. 属性综合法（自底向上法）

      采用属性分析开始，高层实体以及联系通过底层属性组成的设计技术

      实际上，这是一种基于统计分析推导的方法，即通过对数据元素与应用任务联系的定性定量统计分析技术来推导的方法

      统计分析推导的处理过程：可分为属性分类，实体构成，联系的确定等相对独立的步骤

3. 逻辑结构设计

  • 目的（任务）：是将概念模型转化为等价的，并为特定 DBMS 所支持数据模型的结构

  • 数据库逻辑模型表示方法：层次，网状，关系数据模型表示

  • 逻辑结构设计的输入和输出信息

    逻辑结构设计是在需求分析以及概念结构设计的基础上进行的，通常要求提供如下输入信息

    1. 独立于特定 DBMS 的概念模型
    2. 有关响应时间，安全保密性，数据完整性及恢复方面的要求说明，包括保持数据一致性的规则说明
    3. 数据量以及使用频率
    4. 特定 DBMS 特性，包括 DBMS 支持的数据模型以及数据定义语言的说明

  • 完成逻辑结构设计之后输出的信息

    1. 一个特定 DBMS 支持的概念模式，或称为模式
    2. 物理设计说明，主要包括存入数据库中的数据量，使用频率以及响应时间要求
    3. 程序设计说明，其是在需求分析的基础上，根据已完成的逻辑模型编制各程序名，执行逻辑步，存取数据的名，顺序以及操作特征（读写删改）的说明

  • 逻辑结构设计的步骤

    1. 模型转换是指将概念模型等价地转换为特定 DBMS 支持的关系模式，网状模型或层次模型

    2. 子模式设计的目标：是抽取或导出模式的子集，以构造不同用户使用的局部数据逻辑结构

    3. 编制应用程序设计说明的目的：为可实际运行的应用程序设计提供依据与指导，并作为设计评价的基础

    4. 设计评价的任务是分析并检验模式以及子模式的正确性合理性

4. 物理结构设计

  • 概念：是指对于一个给定的数据库逻辑结构，研究并构造物理结构的过程
  • 任务：是确定数据库在存储设备上的存储结构以及存储方法

    因 DBMS 的不同还可能包括建立索引和聚集，以及物理块大小，缓冲区个数和大小，数据压缩的选择等

    对于物理结构设计的人员的要求

    • 掌握基本存储设备的特性
    • 充分了解操作系统以及 DBMS 所提供的支持环境
    • 并十分熟悉数据库的逻辑结构以及存取速度，频率以及数据量方面的要求

5. 数据库实施

  • 工作内容：加载数据，应用程序设计以及数据库运行

  • 数据加载

    概念：将符号要求的数据装载到数据库中去，其具体过程包括数据的收集，分类，整理校验和输入等过程，经此就可将数据按数据库定义准确地存储到数据库中

  • 应用程序设计

    严格说，应用程序设计不应是数据库设计的范畴，但它又必须且只能是随着数据库结构设计的进展而逐步地并行进行

  • 数据库试运行

    目的：为了避免对实际应用带来损害，数据库通常应经过一个试运行阶段

6. 数据库的运行和维护

  • 数据库系统投入实际运行标志着数据库设计和应用开发的基本完成，但绝不是意味着设计应用开发工作的终止
