数据库 - 关系型数据库（三）关系数据库的规范化理论

发布日期 2018-11-04 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 638 | 阅读时间 ≈ 2

为什么要规范化（理论）

  • 关系模型中可能存在的一些问题，
  • 规范化的作用
    1. 是关系数据库设计的理论依据
  • 范围
    1. 是关系模式中各属性之间的依赖关系及其对关系性能的影响，为关系模式提供了优劣的理论标准

关系模型中可能存在冗余和异常问题

  1. 数据冗余：同一数据被反反复复存储的情况

  2. 更新异常：数据冗余将导致存储空间的浪费和潜在数据不一致修改麻烦等问题

  3. 插入异常：应该插入到数据库中的数据不能执行插入操作的情形

  4. 删除异常：不应该删去的数据被删去的情形

    上述的情况和数据依赖密切相关

函数依赖和关键字（抄不动了😢）

  1. 函数依赖：指关系中属性间的对应关系
  2. 定义：设 R 为任一给定关系，如果对于 R 中属性 X 的每一个值，R 中的属性 Y 只用唯一值与之对应，则称 X 函数决定 Y 或称 Y 函数依赖于 X,记住 X -> Y。其中，X 称为决定因素

函数依赖根据不同性质的分类（抄不动了😢）

  1. 完全函数依赖
  2. 部分函数依赖
  3. 传递函数依赖

范式与关系规范化（抄不动了😢）

  1. 规范化：一个低一级范式的关系模式通过模式分解（Schema Decomposition）可以转换为若干个高一级范式的关系模式的集合的过程

范式

  1. 第一范式(1NF)

    • 不含重复组的关系，不存在嵌套结构
    • 不满足第一范式的关系称为非规范关系

  2. 第二范式(2NF)

    • 定义：设 R 为任一给定关系，若 R 为 1NF, 且所有非主属性都完全函数依赖于候选关键字，则 R 为第二范式

  3. 第三范式(3NF)

    • 定义：设 R 为任一给定关系，若 R 为 2NF，且起每一个非主属性都不传递函数依赖于候选关键字，则 R 为第三范式

      通常，第三范式的关系大多数都能解决插入和删除操作异常的情况，数据冗余也能得到有效的控制

  4. BCNF（抄不动了😢）

    关系规范化理论主要应用于数据库设计中的概念设计阶段，用来分析其实体划分是否合适，判断属性具体分配到哪个实体中

变量和字面量

  • 字面量：是原始数据，例如： 字符串是数据本身

PHP 中的数据类型

  • Integer（整数）
  • Float（浮点数，也叫 Double，双精度）
  • String（字符串）：用来表示字符串
  • Boolean（布尔）：用来表示 true 或者 false
  • Array（数组）：用来保存具有相同类型的多个数据项
  • Object（对象）：用来保存类的实例

  • 三种特殊的类型：

    1. NULL（空）

      没有被赋值，已经被重置或者被赋值的特殊值 NULL 的变量就是 NULL 类型变量

    2. resource（资源）：特定的内置函数（例如数据库函数）将返回 resource 类型的变量

      基本上不能直接操作一个 resource 变量，但是通常他们都将被函数返回，而且必须作为参数传递给其他函数

    3. callable

      通常都是可以传递给其他函数的函数

    JavaScript 则不太一样😁
    分为两大类：1. 原始类型，2. 引用类型

    1. 原始类型（6种）：Boolean，Null，Undefined，Number，String，Symbol (ECMAScript 6 新定义)
    2. 引用类型（object）: Object, Array，Map, Set, Date, RegExp, Math, HTMLBodyElement,HTMLElement, Element, Node, EventTarget，等等各种 Element Object

类型强度

  • PHP 是一种弱类型的语言，或者动态类型语言（这个和 JavaScript 基本一致）

    变量的类型是由赋给变量的值确定的，自动地获取输入的数据类型

类型转化

  • 使用类型转化，可以将一个变量或值转换成另一种类型（和 C 语言的类型转换是相同的）

可变变量(🐂)

  • PHP 中还有其他类型的变量：可变变量

    允许我们动态地改变一个变量的名称

声明和使用常量

  • 常量：一旦设定之后，在脚本的其他地方就不能再更改(常量名称通常由大写的字母组成)

    使用的时候直接不需要在常量名前添加 $

理解变量作用域(待完善)

  • 作用域：是指在一个脚本中某个变量在那些地方可以使用或可见

  • PHP 定义了 6 条规则：

    1. 内置超级全局变量可以在脚本的任何地方使用和可见（和 JavaScript 类似）,例如：

      $GLOBALS

      $_SERVER 服务器环境变量数组

      $_GET

      $_POST

      $_COOKIE

      $_FILES

      $_ENV

      $_REQUEST

    2. 常量，一旦被声明，将在全局可见；也就是说，它们可以在函数内外使用

    3. 在一个脚本中声明的全军变量在整个脚本中是可见的，但不是在函数内部（和 JavaScript 类似）

    4. 函数内部使用的变量声明为全局变量时，其名称要与全局变量名称一致（和 JavaScript 类似）

    5. 在函数内部创建并声明为静态的变量无法在函数外部可见，但是可以在函数的多次执行过程中保持该值

      什么是静态变量？
      前半句明白，后半句感觉和闭包类似

    6. 在函数内部创建的变量对函数来说是本地的，而当函数终止时，该变量也就不存在了（和 JavaScript 类似，函数运行结束之后，变量也随即被销毁）

数据类型 (Data Type)

  • INTEGER：存储整数的数据类型（无法存储小数）
  • CHAR：（定长字符串）
    1. 在括号中指定该列可以存储的字符串的长度（最大长度），超过的部分无法输入到该列中
    2. 若存储的长度没有到达指定的长度，剩余的长度使用 半角空格保存
    3. 例子，Data Type: CHAR(6)，若数据写入只有 ‘james’，实际在数据库中的数据：’james ‘
  • VARCHAR：（可变字符串）
    1. 在括号中指定该列可以存储的字符串的长度（最大长度, 超过的部分无法输入到该列中??）
    2. 输入写入的长度就是实际存储的长度
    3. 例子，Data Type: CHAR(6)，若数据写入只有 ‘james’，实际在数据库中的数据：’james’
  • NUMERIC
    1. 用来存储（全体位数，小数位数）的形式来指定数值的大小
  • DATE：用来指定存储日期（年月日）的列的数据类型（日期型）

数据库 - 数据库系统（三）数据模型

发布日期 2018-10-27 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 3.1k | 阅读时间 ≈ 10

模型（P37）

1.（是什么？）是现实世界特征的模拟和抽象表达
2.（作用：）它有助于人们更好地认识和理解客观世界中的事物，对象，过程等感兴趣的内容

数据模型（P37）

  1. 数据模型用来模拟和抽象现实世界中的数据特征
  2. 数据模型也是一种模型，它是对现实世界数据特征的抽象，描述的是数据的共性内容

数据特征（P37）

  1. 数据特征：静态和动态

    静态特征：包括数据的基本结构，数据间的联系以及对数据取值范围的约束
    动态特征：指对数据可以进行符合一定规则的操作。

    所以，对现实世界的数据特征进行抽象的数据模型，需要描述数据的静态特征和动态行为，并为数据的表示和操作提供框架

  2. 终于要回答数据模型是什么了：
    数据模型是用来描述数据的结构（静态特征），定义在结构上的操纵（动态行为）以及数据间的约束的一组概念和定义

数据模型组成要素（P37）

  1. 数据模型三个要素：
    • 数据结构
    • 数据操纵
    • 数据约束

  1. 数据结构
    数据结构描述的是系统的静态特性，即数据对象的数据类型，内容，属性以及数据对象之间的联系。

    数据结构反映了数据模型最基本的特征，因此人们常常按照数据结构的类型来命名数据模型：

    • 层次结构 -> 层次模型
    • 网状结构 -> 网状模型
    • 关系结构 -> 关系模型

  1. 数据操作
    数据操作描述的是系统的动态特性，是对各种对象的实例允许执行的操作的集合

    • 数据操作分类：更新和检索

      其中更新：插入，删除和修改

      数据模型必须定义这些操作的确切含义，操作符号，操作规则（如优先级）以及实现操作的语言

  2. 数据约束

    • 数据约束（是什么？）是描述数据结构中数据间的语法和语义关联，包括互相制约与依存关系以及数据动态变化规则，以保证数据的正确性，有效性与相容性

    • 数据约束包括：

      1. 数据完整性约束
      2. 数据安全性约束
      3. 并发控制约束

    • 数据约束既刻画了数据静态特征，也表示了数据动态行为规则

数据模型的分类（P38）

  • 数据模型满足的三个条件：

    1. 能比较真实地模拟现实世界
    2. 容易为人们所理解
    3. 便于在计算机上实现

  • 为了把现实世界中的对象转化为数据管理系统中所支持的数据结构采用的方法： 逐步抽象的方法：

    1. 在不同的抽象层使用不同的数据模型，从而将实现世界中的具体事物逐步抽象，转换，组织成机器世界（计算机）中某一具体数据库管理系统所支持的数据类型。

      1. 首先，将实现世界中的客观对象抽象为信息世界汇总的某一种信息结构，这种信息结构并不依赖于具体的计算机系统，也不与具体的数据库管理系统相关，是一种概念性的模型
      2. 再将概念级的模型转换为机器世界中某以具体的数据库管理系统支持的数据模型

数据模型：概念层数据模型

  1. 概念层是数据抽象级别的最高层，

    其目的是按用户的观点来对世界建模

  2. 概念层数据模型，也称为数据的概念模型（Conceptual Model）或信息模型

    它是描述现实世界的事物，与具体的计算机系统无关，且独立于任何 DBMS，但容易向 DBMS 所支持的逻辑数据模型转换
    这类数据模型主要用于数据库的设计阶段，即在设计数据库时，通常用概念模型来抽象，表示现实世界的各种事物及其联系

信息世界的基本概念

  1. 关于概念模型：
    • 概念模型用于信息世界的建模
    • 是实现世界到信息世界的第一层抽象
    • 是数据库设计人员进行数据库设计的得力工具
    • 也是数据库设计人员和用户之间进行交流的语言
  1. 概念模型特点：
    • 一方面，概念模型具有较强的语义表达能力，能够方便，直接地表达应用中的各种语义知识
    • 另一方面，它简单，清晰，易于用户理解
  1. 基本概念：
    • 实体（Entity）: 客观存在并可相互区别的事物称为实体。

      例如，商品，学生，课程，班级

    • 属性（Attribute）: 实体所具有的某种特性称为实体的属性。

      例如，实体可以拥有多个属性，如商品的名称，商品的介绍，商品的价格

    • 码或键（Key）：可唯一标识实体的属性集。

      例如，商品的名称是商品的码或键

    • 域（Domain）：属性的取值范围。

      例如，商品的名称的域，为字符集合，也有长度限制

    • 实体型（Entity Type）：类似的实体可能拥有共同的特质和性质，用实体名和属性名集合来抽象和刻画同类实体，称为实体型，

      例如，商品（名称，编号，价格，成本，生产日期）就是一个实体型

    • 实体集（Entity Set）：同型实体的集合称为实体集。每个商品是一个实体，所有的商品构成一个实体集
    • 联系（Relationship）：在现实世界中，事物内部以及事物之间是有联系的，这些联系在信息世界中反映为实体（型）内部的联系和实体（型）之间的联系

      实体内部的联系通常是指视图各属性之间的联系。
      例子1，每个商品生产的时候会产生一个唯一的编号，只要知道这个编号就可以找到对应的商品，获得商品的其他属性
      例子2，每40个学生组成一个班级，学生这个实体型和班级这个实体型之间存在联系

概念模型的表示方法（E-R 图）

  1. 实体-联系方法（Entity-Relationship approach）该方法用 E-R 图（E-R diagram）（用来描述现实世界的概念模型）

    E-R 图：提供了表示信息世界中实体，属性和联系的方法，其中实体，属性，联系的图形含义分别如下

    1. 实体型，其用矩形表示，矩形框内写明实体的名称
    2. 属性，其用椭圆形表示，并用无向边将其与相应的实体连接起来
    3. 联系，其用菱形表示，菱形框内写明联系的名称，并用无向边分别与有关实体联系起来

    E-R 图的特点：

    1. 实体-联系方法是抽象和描述实现世界的有力工具。
    2. 用 E-R 图表示的概念逻辑模型独立与具体的 DBMS 所支持的逻辑模型，它是各种逻辑模型的共同基础，比逻辑模型更一般，更抽象，更接近实现世界

逻辑层数据模型（数据的逻辑模型）

  1. 逻辑层是数据抽象的中间层，描述数据整体的逻辑结构。

    它是用户通过数据库管理系统看到的现实世界，是基于计算机的观点对数据进行建模和表示
    它既要考虑用户容易理解，又要考虑便于 DBMS 实现
    任何 DBMS 都是基于某种逻辑数据模型
    主要的逻辑数据模型：

    1. 层次模型
    2. 网状模型
    3. 关系模型（重点）
    4. 面向对象模型

  2. 层次模型

    1. 层次模型是数据库管理系统最早使用的一种数据模型，它的数据结构是一棵 “有向树”，树的每个结点对应一个记录集，也就是现实世界的实体集。

    2. 特点（优点）:

      • 有且仅有一个结点没有父节点，它被称作根节点，其他的节点有且仅有一个父结点。例如，组织机构的图谱就是典型的层次结构

    3. 缺点：

      • 现实世界中的关系是多样的，层次模型难以表达实体之间这个复杂的联系

  3. 网状模型

    1. 网状结构以网状结构表示实体与实体之间的联系。
    2. 特点（优点）：
      • 网状模型和层次模型之间的关系：网状模型是层次模型的扩展，允许节点有多于一个父结点
      • 现实世界的关系很复杂，网状模型可以方便地表示各种类型之间的联系，即可以表示从属的关系，也可以表示数据间的交叉关系
    3. 缺点：
      • 结构复杂，实现的算法难以规范化

  4. 关系模型

    1. 关系模型是用二维表结构来表示实体以及实体间联系的模型，并以二维表格的形式组织数据库中额数据
    2. 特点（优点）：
      • 关系模型是建立在严格的数学概念的基础上
      • 关系模型的概念单一，统一用关系来表示实体以及实体之间的联系，对数据的检索和更新结果同样也是用关系（即表）来表示，

        关系模型的数据结构简单，清晰，用户易懂，易用

      • 关系模型的存取路径对用透明，从而具有更高的数据独立性，更好的安全保密性，也简化了程序元的工作和数据库建立的工作

        目前商用数据库多是关系型数据库，例如：MySQL 就是一个关系数据库管理系统

  5. 面向对象模型

    1. 面向对象方法与数据库结合所构成的数据模型称为面向对象模型（面向对象模型既是概念模型又是逻辑模型），面向对象数据模型用面向对象观点来描述现实世界实体的逻辑组织，对象间的联系
    2. 特点：
      • 其表达能力丰富，具有对象可复用，维护方便的优点
      • 是正在发展的数据模型，也是数据库的发展方向之一

物理层数据模型

  1. 物理层数据模型，也称为数据的物理模型（Physical Model）

    其描述数据在存储介质上的组织结构
    是逻辑模型的物理实现，即每一种逻辑模型在实现时都有与其相对应的物理模型

  2. 物理模型是数据库最底层的抽象，它确定数据的物理存储结构，数据存取路径以及调整，优化数据库的性能，
  3. 物理模型的设计目标，是提高数据库性能和有效利用存储空间。
  4. 影响物理层数据模型的因素：
    • 不但由 DBMS 的设计决定，而且与操作系统，计算机硬件密切相关。

      物理数据结构一般都向用户屏蔽，用户不必了解其细节

概念模型，逻辑层数据模型以及物理层数据模型之间的关系

  1. 这三个不同的数据模型既相互独立，而又存在着关联
    1. 从现实世界到概念模型的转换是由数据库设计人员完成
    2. 从概念模型到逻辑模型的转换可以由数据库设计人员完成，也可以用数据库设计工具协助设计人员完成
    3. 从逻辑模型到物理模型的转换主要是由数据库管理系统完成

数据库 - 数据库系统（二）数据库的结构

发布日期 2018-10-25 | In 书籍笔记 , 《数据库系统原理》(04735)
文章总字数: 1.5k | 阅读时间 ≈ 5

数据库系统的结构

数据库系统的分类（P33）

  1. 数据库管理员（DBA）的角度

    1. 内部系统
      • 通常采用三级模式结构
    1. 外部系统
      • 通常采用集中式结构，分布式结构和并行结构等

  2. 从用户的角度

    1. 客户/服务器结构
    2. 浏览器/服务器结构

数据库系统的三级模式结构

  • 三级模式结构（采用三级模式结构，并提供两层映像功能）：

    1. 用户级
    2. 概念级
    3. 物理级

  • 具体：

    1. 模式
    2. 外模式
    3. 内模式

模式（P34）

  • 模式也称为概念模式或逻辑模式

    它是数据库中全体数据的逻辑结构和特征的描述，是所有用户的功能数据视图

  • 模式实际上市数据库在逻辑级上的视图，概念视图

    特点：

    1. 其形式要比数据的物理存储方式抽象

    2. 不包含任何数据库的实现细节，而是定义数据的逻辑结构，数据之间的联系，以及数据相关的安全性，完整性(DBA 必须考虑整个数据库的这种全局概念视图)

    3. 处于数据库外模式和内模式的中间，一个数据库只有一个模式

外模式（P34）

  • 三级模式结构中，外模式也称为子模式（Subschema）或用户模式
  • 是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据和逻辑结构和特征的描述
  • 是某一应用有关的数据的逻辑表示
  • 实际上：用于满足不同数据库用户需求的数据视图，即用户视图。通常是模式的子集，是对数据整体数据结构的局部重构
  • 一个数据库可以有多个不同的外模式，允许它们有一定的重叠、

    DBMS 提供子模式描述语言（Subschema DDL）来严格定义子模式

内模式（P35）

  • 在三级模式结构中，内模式也称为存储模式（Storage Schema）
  • 它是对数据库中数据物理结构和存储方式的描述，是数据在数据库内部的表示形式
  • 实际上：是整个数据库的最底层表示，它不同于物理层，是 DBA 所见的，特定的 DBMS 所处理的数据库的内部结构，即内部视图或存储视图
  • 一个数据库只用一个內模式

    內模式不是存储设备上的物理记录或物理块，也不涉及具体设备限制，内模式还不是最底层的物理层，比内模式更接近物理存储和访问的那些软件机制是操作系统的一部分

    DBMS 提供內模式描述语言（內模式 DDL, 或存储模式 DDL）来严格定义内模式

    1. 定义内部记录类型，索引和文件的组织方式
    2. 以及所有数据控制方面的细节（例如：记录存储方式是按照顺序结构存储还是按照 B 树结构存储）
    3. 数据是否压缩，加密等

三级模式结构的两层映像与数据独立性（P35）

  • 数据库系统三级模式结构的三个模式分别是对数据的三级抽象，它们之间的关系

    特点：

    1. 一个数据库的整体逻辑结构和特征的描述（概念模式）是独立于数据库其他层次结构（内/外模式）的描述，是数据库的核心，也是数据库设计的核心
    2. 一个数据库的内部存储模式依赖于概念模式，但存储模式独立于外部模式，也独立于具体的存储设备
    3. 用户逻辑结构（外模式）是在全局逻辑结构描述的基础上定义的，它面向具体的应用程序，独立于内部模式和存储设备
    4. 特定的应用程序是在外模式的逻辑结构上编写，它依赖于特定的外模式，与数据库的模式和存储结构独立
  • 为了有效支持数据的三级抽象以及他们相互间的联系和转换，DBMS 通过在内部提供三级模式之间的两层映像来实:
    1. 外模式/模式映像
    2. 模式/内模式映像
外模式/模式映像（P35）
  1. 同一个模式可以有任意多个外模式
模式/内模式映像（P36）
  1. 模式/内模式映像是唯一的

客户/服务器结构（Client/Server, C/S）（P36）

  1. 客户端（前台， 表示层）

    数据库的使用者与命令行客户端，图形化界面管理工具，应用程序等直接交互，而不与数据库直接联系，这种结构中，命令行客户端，图行化界面管理工具，应用程序等称为 “客户端”(“前台” 或 “表示层”)

    主要完成与数据库使用者的交互任务

  2. 服务器（后台，数据层）

    数据库管理系统则称为”服务器”，”后台”，”数据层”

    主要负责数据管理

浏览器/服务器结构（Browser/Server, B/S）（P37）

  1. 浏览器/服务器结构是一种基于 Web 应用的 客户/服务器结构，也称三层客户/服务器结构

    在数据库系统中，它将数据库管理系统交互的客户端进一步细分为 “表示层” 和 “处理层”

    1. “表示层”：是数据库使用者的操作和展示界面，通常由浏览器构成
    2. “处理层”：也称为 “中间层”，主要负责处理服务器使用者的具体应用逻辑
